# 한이음 프로젝트 중간 발표 핵심 정리

> 팀 공유용 브리프

## 문서 목적

중간 발표 전 조원들이 반드시 공유해야 할 내용을 한 문서로 정리한다.  
범위는 UX 설계, 앱 출시 준비, Open API, OCR 구현, DB/AI 연동 구조, 개발 과정의 어려움, 예상 질문과 답변이다.

## 1. 프로젝트 핵심 개념

본 프로젝트는 **시각장애인 사용자가 전자책 앱에서 원하는 도서를 더 쉽게 찾을 수 있도록 돕는 보조 앱**을 구현하는 것이 핵심이다.

앱은 다음 흐름을 연결한다.

- 음성 검색
- ISBN 탐색
- 접근성 API 기반 앱 제어
- 화면 캡쳐 및 OCR
- OCR 결과 필터링/구조화
- TextToSpeech 기반 음성 안내

핵심 사용 시나리오는 다음과 같다.

1. 보호자의 도움으로 오버레이, 접근성, 화면 캡쳐, 마이크 권한을 초기 활성화한다.
2. 사용자는 음성으로 찾고 싶은 도서를 말한다.
3. AI/서버는 도서명을 추출하고 ISBN을 찾는다.
4. 앱은 접근성 API로 대상 전자책 앱 검색창에 ISBN을 입력한다.
5. 검색 결과 화면을 캡쳐하고 OCR로 인식한다.
6. 검색 결과를 필터링한 뒤 사용자에게 음성으로 안내한다.

## 2. UX 설계 흐름

```text
보호자의 도움으로 오버레이 및 접근성 권한 활성화
↓
앱 실행
↓
음성으로 찾고 싶은 도서 검색
↓
AI가 해당 도서의 ISBN 코드를 DB 또는 Open API 기반 데이터에서 탐색
↓
AI가 얻은 ISBN 코드를 앱 내부 검색창에 입력하고 검색 버튼 클릭
↓
검색 결과를 AI가 OCR로 인식 및 필터링
↓
사용자에게 원하는 도서 정보를 음성으로 안내
```

## 3. 앱 출시 이전 절차

| 단계 | 핵심 작업 | 발표 시 강조점 |
|---|---|---|
| 개발자 등록 | 개발자 계정 유형은 개인으로 선택하고, 등록비 $25를 결제한다. 비용 신청은 7월 초 한이음 사무국 신청 기간에 진행한다. | 등록비와 계정 생성은 출시 전 선행 조건이다. |
| Play Console 준비 | 개발자 정보 입력 및 인증 후 Play Console에 접속해 앱을 생성한다. | 앱 기본 정보와 권한 설명을 사전에 준비해야 한다. |
| 권한 설명 | 접근성 서비스, 화면 캡쳐, 마이크 권한 사용 목적을 작성한다. | 시각장애인 대상 접근성 보조 목적을 명확히 설명해야 한다. |
| 테스트 트랙 | 12명의 테스터가 14일간 설치 및 사용하도록 테스트 트랙을 설정한다. | 프로덕션 액세스 신청 전 필수 검증 단계다. |
| 프로덕션 신청 | 프로덕션 액세스 신청 후 프로덕션 릴리즈를 제출하고 심사에 대응한다. | 접근성 및 개인정보처리방침 관련 질문에 대비해야 한다. |

### 출시 유의점

- 개인정보처리방침에는 접근성, 화면 캡쳐, 마이크 권한 사용 목적이 포함되어야 한다.
- 앱 내부에도 사용자가 이해할 수 있는 권한 설명 화면을 두어야 한다.
- 프로덕션 액세스 신청 시 **시각장애인 대상 접근성 보조 목적**을 명확히 설명해야 한다.

## 5. OCR 로직 구현 현황

현재 구현은 Kotlin 기반이다.  
오버레이 서비스에서 OCR 실행을 시작하고, 화면 캡쳐, crop, ML Kit Korean OCR, 후처리 분석기를 거쳐 결과를 다시 오버레이에 표시하는 구조다.

### 현재 OCR 처리 흐름

```text
EbookOverlayService.kt에서 '책 후보 찾기' 버튼 클릭
↓
hideOverlay()로 OCR 전에 오버레이 숨김
↓
오버레이 글자까지 OCR로 읽히는 문제 방지
↓
좌우 4%, 상단 6%, 하단 6% 제외 화면 일부만 OCR 대상으로 설정
↓
ScreenCaptureService에 캡쳐 요청
↓
crop 영역과 현재 대상 앱 패키지를 전달
↓
ScreenCaptureService.kt에서 OCR 대상 영역 crop
↓
OcrProcessor.kt에서 ML Kit Korean OCR 실행
↓
ScreenCaptureService.kt에서 OCR 결과를 후처리기로 전달
↓
SearchOcrAnalyzer.kt에서 OCR 결과를 줄 단위로 정렬
↓
각 줄의 화면 위치 저장
↓
지원 앱인지 확인
↓
targetPackage 기준으로 앱별 후처리 조건 적용
↓
책 후보만 필터링
↓
출력 형식 생성
↓
ScreenCaptureService.kt에서 결과를 오버레이로 다시 표시
```

### 앱별 후처리 조건

| 지원 앱 | 후처리 조건 |
|---|---|
| 교보문고 | 제목 아래에 자주 나오는 설명 문장을 차단한다. |
| 알라딘 / YES24 | 제목이 길게 잘려 줄바꿈되는 경우를 대비해 이어 붙인다. |

### 업데이트 방향

기존 규칙 기반 OCR 후처리 코드는 폐기하거나 리팩토링할 예정이다.  
이후에는 NVIDIA AI에게 판단을 위임해 OCR 정확성과 앱별 UI 대응력을 높이는 방향으로 전환한다.

## 7. DB 구축 계획

DB는 **Aiven + MySQL** 조합을 사용할 예정이다.

### 구축 방식

- Python으로 Open API를 호출해 필요한 도서 데이터를 수집한다.
- 기본 레코드 구조는 다음과 같다.

| 필드 | 설명 |
|---|---|
| 책 제목 | 검색 및 사용자 안내에 사용 |
| 저자 | 동명이서/유사 제목 구분에 사용 |
| 출판사 | 검색 결과 검증 및 안내에 사용 |
| ISBN 코드 | 전자책 앱 검색창 입력에 사용 |

### 검토할 점

모든 e-book 데이터를 자체 DB에 저장하는 방식은 신중하게 검토해야 한다.

- API 호출 제한이 있다.
- 과도한 호출은 비정상 트래픽으로 판단될 수 있다.
- API로 제공되는 도서 정보는 서비스 제공자의 자산이다.
- 자체 DB 구축과 기존 API 활용 사이의 실제 시간 차이가 크지 않을 수 있다.

## 8. 음성 검색부터 음성 안내까지의 통합 구조

### 주요 구성 요소

| 구성 요소 | 역할 |
|---|---|
| `SpeechRecognizer` | 사용자 음성을 텍스트로 변환한다. `RECORD_AUDIO` 권한이 필요하다. |
| `FastAPI` 서버 | NVIDIA AI 호출, 도서명 추출 및 정규화, DB/API 기반 ISBN 조회를 담당한다. |
| NVIDIA LLM | 도서명 추출, 정규화, OCR 결과 구조화, 제어 판단에 사용한다. |
| `AccessibilityService` | 접근성 트리 탐색, `ACTION_SET_TEXT`, `ACTION_CLICK`, `dispatchGesture` fallback을 수행한다. |
| `MediaProjection` | 화면 캡쳐/화면 녹화 권한을 통해 검색 결과 화면을 확보한다. |
| ML Kit Korean OCR | 캡쳐 이미지에서 한국어 텍스트를 추출한다. |
| `TextToSpeech` | 사용자에게 검색 결과와 선택지를 음성으로 안내한다. |

### 통합 처리 흐름

```text
사용자 음성 검색
↓
SpeechRecognizer가 텍스트로 변환
↓
서버가 NVIDIA AI 호출
↓
AI가 책 제목 추출
↓
AI가 책 제목 정규화
↓
DB에서 ISBN GET
↓
ISBN이 여러 개일 경우 TTS로 사용자의 선택 유도
↓
대상 앱 확인
↓
접근성 트리로 검색창 찾기
↓
검색창에 ISBN 입력
↓
검색 버튼 클릭
↓
화면 캡쳐
↓
ML Kit OCR로 텍스트 추출
↓
서버로 OCR 텍스트 전송
↓
NVIDIA AI가 OCR 결과 구조화
↓
서버가 구조화된 결과를 앱으로 전송
↓
TextToSpeech로 음성 안내
```

### OCR 보강 옵션

OCR 전용 보강이 필요할 경우 `nvidia/nemotron-ocr-v2`를 추가로 검토한다.

## 9. 개발 과정에서의 어려움

| 문제 | 구체적 상황 | 대응 방향 |
|---|---|---|
| 실시간 OCR 검증의 어려움 | 에뮬레이터 속도가 낮아 검증이 느렸다. 공기계 테스트를 진행해도 미출시 앱 특성상 수정 코드를 실시간으로 반영하기 어려웠고, 동료가 GitHub에서 매번 수정 코드를 내려받아야 했다. | 공기계 테스트 절차를 정리하고, 테스트 APK 공유 및 로그 수집 흐름을 단순화한다. |
| 낮은 OCR 후처리 성능 | 교보문고를 제외한 나머지 앱은 UI가 복잡해 규칙 기반 후처리만으로 준수한 성능을 확보하기 어려웠다. | 규칙 기반 후처리 중심에서 LLM 기반 판단으로 전환한다. |

## 10. 예상 질문 및 답변

### Q1. 앱을 제어하는 AI로 NVIDIA LLM을 선택한 이유는?

Claude, Codex 등 주요 상용 모델은 종량제 비용 부담이 우려된다.  
따라서 무료로 공개된 오픈 소스 모델을 우선 검토해야 했다.

후보로는 다음 모델을 고려했다.

- Qwen
- DeepSeek
- Gemini
- NVIDIA `nvidia/llama-3.1-nemotron-nano-8b-v1`

최종적으로 최근 개발자 커뮤니티에서 주목받은 NVIDIA `nvidia/llama-3.1-nemotron-nano-8b-v1`을 선택했다.

다만 AI가 전체적인 제어 판단을 보조할 뿐, 세부 조작은 Android의 무료 API를 사용한다.  
따라서 모델 선택에 따른 유불리는 제한적일 것으로 예상한다.

향후 Android 서비스와의 호환성이 더 중요해질 경우 Gemini로 교체하는 것도 고려한다.  
또한 코드는 통합 제어용 AI를 언제든 교체할 수 있는 형태로 작성할 예정이다.

### Q2. 모든 e-book 데이터를 불러오는 것이 가능한가?

현실적으로 어렵다고 본다.

이유는 다음과 같다.

- 알라딘 API는 일일 호출 수가 15,000회로 제한되어 있다.
- 너무 빈번한 API 호출은 서비스 제공자에게 비정상 트래픽으로 판단될 수 있다.
- API 키를 발급받은 계정이 정지될 위험이 있다.
- API로 호출할 수 있는 정보 자체가 해당 회사의 자산이다.
- 외부 DB에 사실상 복사본 수준으로 존재하는 것을 서비스 제공자가 용납할 가능성은 낮다.

### Q3. DB 구축은 필수적인가?

필수는 아니라고 본다.

도서 앱 API를 활용해 모든 데이터를 수집하고 자체 DB를 구축할 수 있다고 가정해도, 자체 DB 검색과 기존 API 활용 사이의 시간 차이가 크지 않을 가능성이 있다.

따라서 DB는 다음 목적 중심으로 검토하는 것이 적절하다.

- 테스트 편의성
- 검색 안정성
- 자주 쓰는 데이터 캐싱
- API 장애 또는 응답 지연에 대한 보완

### Q4. 도서 앱 위에서 보조 앱을 구동하고, 그 도서 앱을 조작하는 것이 보안상 허용되는가?

현재까지의 조사 기준으로는 가능하다고 본다.

Android는 iOS와 달리 `AccessibilityService` API를 제공한다.  
사용자가 권한을 허용한 경우, 특정 앱을 다른 앱이 부분적으로 조작할 수 있다.

다만 다음 제한이 있다.

- 앱별로 실제 허용되는 조작 범위는 테스트 전 알 수 없다.
- 권한이나 조작이 제한되는 앱은 지원 대상에서 제외해야 한다.
- 접근성 서비스를 사용하는 앱은 출시 심사에서 개인정보처리방침 준수 여부를 엄격하게 검토받을 수 있다.
- 개발 중 또는 출시 전 단계에서 잠재적 심사 리스크가 될 수 있다.

## 11. 발표 전 체크리스트

- [ ] 프로젝트 목적을 **시각장애인 대상 전자책 검색 보조**로 일관되게 설명한다.
- [ ] 권한 사용 이유를 접근성 보조 목적, 화면 검색 결과 인식, 음성 입력/안내로 분리해 말한다.
- [ ] OCR 후처리 한계와 LLM 전환 이유를 기술적 필요성 중심으로 설명한다.
- [ ] DB는 필수 구성요소가 아니라 검색 안정성과 테스트 편의성을 위한 선택지라고 정리한다.
- [ ] NVIDIA 모델은 비용, 오픈 소스 접근성, 교체 가능한 구조라는 관점에서 설명한다.
- [ ] Play Console 심사 리스크는 개인정보처리방침과 앱 내부 설명 화면으로 대응한다고 말한다.

## 12. 발표에서 특히 강조할 메시지

> 이 앱은 전자책 앱을 대체하는 서비스가 아니라, 시각장애인이 기존 전자책 앱을 더 쉽게 사용할 수 있도록 돕는 접근성 보조 앱이다.

발표에서는 다음 세 가지 메시지를 반복해서 잡아주는 것이 좋다.

1. **접근성 목적**: 시각장애인 사용자의 도서 검색 과정을 보조한다.
2. **기술적 현실성**: Android 접근성 API, OCR, TTS, Open API를 조합해 구현 가능성이 있다.
3. **리스크 인식**: 권한 심사, 앱별 조작 제한, OCR 정확도, API 제한을 알고 있으며 대응 방향을 마련하고 있다.
